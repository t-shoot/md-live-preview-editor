# フロントマター表示 要件定義

## 概要
Markdown Live Preview は現在、文書先頭の YAML フロントマター(`---` で囲まれたメタデータ)を
一切特別扱いしておらず、生の `---` 行と YAML テキストがそのまま表示される。
VS Code 純正の Markdown プレビュー(`markdown-language-features`)は、フロントマターを
解析してキー・値のテーブルとして表示し、解析に失敗した場合はエラー表示に切り替える。
本機能は、この純正プレビューと同等の「フロントマターのテーブル表示」を実現する。

参考(VS Code 純正プレビューの実装を直接確認した事実。design 段階の入力とする):
- フロントマターは文書の**先頭行**が `---` で始まり、以降で単独の `---` 行が現れるまでの範囲。
- 解析成功かつキーが1件以上ある場合: `<table class="frontmatter"><tbody><tr><th>キー</th><td>値</td></tr>...</tbody></table>` 形式で表示。
- 解析成功だがキーが0件(空フロントマター)の場合: 何も表示しない。
- 解析失敗の場合: 「Failed to parse frontmatter」+ エラー内容を表示。
- ライト/ダーク/ハイコントラストいずれの VS Code テーマでも配色が切り替わる。

対象外(本要件定義のスコープ外。VS Code 純正プレビューにもない、または本アプリのアーキテクチャに
そぐわないため今回は扱わない):
- ソースとプレビューを別ペインで同期させる差分ハイライト(`.code-line` 系)。本アプリは単一ペインの
  WYSIWYG編集のため、別ペイン同期という前提自体が存在しない。

## 要件

**REQ-1 (Ubiquitous)**
システムは常に、Markdown 文書の 1 行目が `---` のみであり、かつそれ以降に `---` のみからなる行が
存在する場合、その2行に挟まれた範囲をフロントマターブロックとして認識しなければならない。

**REQ-2 (Ubiquitous)**
システムは常に、文書の 1 行目以外の位置に現れる `---` で囲まれた範囲を、フロントマターとして
扱わず通常の Markdown 内容(水平線・見出し等)として処理しなければならない。

**REQ-3 (Event-driven)**
フロントマターブロックを含む文書を開いた、または編集して内容が変化したとき、システムはその
ブロックを YAML として解析し、キーと値の対応表として表示しなければならない。

**REQ-4 (State-driven)**
フロントマターの解析結果が 0 件(トップレベルのキーが存在しない)である間、システムは
テーブルを表示してはならない(何も描画しない)。

**REQ-5 (Unwanted)**
もしフロントマターブロックの YAML 解析に失敗したら、システムは解析エラーであることと
エラー内容を、通常のフロントマターテーブルの代わりに表示しなければならない。

**REQ-6 (Ubiquitous)**
システムは常に、フロントマターの値が配列またはマップ(入れ子オブジェクト)である場合、
その内容を人間が読める形式で表示しなければならない。

**REQ-7 (Optional)**
ユーザーがフロントマター表示領域にカーソルを合わせた場合に限り、システムは元の
YAML ソースを編集可能な状態で提示できる(本アプリは編集可能なライブプレビューであるため、
表示専用の VS Code 純正プレビューとは異なり編集経路を維持する)。

**REQ-8 (Ubiquitous)**
システムは常に、フロントマターテーブルの配色を、既存の CSS テーマ機構
(`mdLivePreview.enabledStyles` で選択中のテーマ、および VS Code のライト/ダーク/
ハイコントラストテーマ)と整合する形で描画しなければならない。

## 受け入れ基準

- AC-1 (REQ-1, REQ-2): `---` で始まり `---` で終わる YAML ブロックが文書の 1 行目にある場合のみ
  フロントマターとして認識される。文書途中に同形のブロックがあってもフロントマター扱いされず、
  従来通り Markdown として解釈されることを確認する。
- AC-2 (REQ-3): `title: テスト` のような単純なキーを含むフロントマターを書くと、
  「title」「テスト」を含む表が表示されることを確認する。
- AC-3 (REQ-4): `---\n---` のように中身が空のフロントマターでは、何も表示されない
  (空のテーブル枠なども出ない)ことを確認する。
- AC-4 (REQ-5): 意図的に壊れた YAML(例: インデント不整合)を書くと、テーブルの代わりに
  エラー表示が出ることを確認する。
- AC-5 (REQ-6): 配列値(`tags: [a, b, c]`)およびネストしたマップ値を持つフロントマターで、
  表内にその内容が読める形で表示されることを確認する。
- AC-6 (REQ-7): フロントマター表示部分にカーソルを合わせる/クリックすると、元の YAML
  ソースが編集できる状態に戻ることを確認する。
- AC-7 (REQ-8): 有効なCSSテーマや VS Code のテーマを切り替えたとき、フロントマターテーブルの
  配色もそれに追従することを確認する。

## 未確定事項(design 段階で決定する)

- YAML 解析用のライブラリ選定(現在 `package.json` に YAML パーサーの依存はなく、新規追加が
  必要。CLAUDE.md の依存追加ルールに従い、design.md で選定理由を明記すること)。
- 表示方式: CodeMirror の decoration/widget として埋め込むか(既存の Mermaid ブロック
  (`mermaidWidget.ts`)と同様の方式が有力)、他の方式にするか。
- REQ-7 のカーソル/クリック時の具体的な相互作用(既存のライブプレビュー要素と同じ操作感に
  揃えるか等)。
